# MyHome LoanPlan — 住宅ローン比較シミュレーター

## 概要
住宅ローンの借入条件から月々の支払額をリアルタイムにシミュレートし、複数の物件を横並びで比較できる Web アプリ。これから住宅購入を検討する個人ユーザーが、物件ごとの実質的な月々負担を直感的に把握し、購入判断を行えるようにすることを目的とする。

## 計算ロジック仕様（受け入れ基準の根拠）

すべての計算関数は、入力が未定義・NaN・Infinity のいずれかを含む場合に `null` を返し、UI 側は '—' を表示する。

1. **資金構成（総借入額）**: `物件価格 + 諸経費`
2. **通常返済元金**: `総借入額 − ボーナス元金`
3. **通常月返済額（元利均等返済）**:
   - 月利 `r = 金利(%) / 100 / 12`、返済回数 `n = ローン期間(年) × 12`
   - `r > 0` のとき: `元金 × r / (1 − (1+r)^(−n))`
   - `r = 0` のとき: `元金 / n`
4. **ボーナス払い（半年ごと元利均等返済）**:
   - 半年利 `r6 = 金利(%) / 100 / 2`、ボーナス返済回数 `m = ローン期間(年) × 2`
   - `r6 > 0` のとき: `ボーナス元金 × r6 / (1 − (1+r6)^(−m))`
   - `r6 = 0` のとき: `ボーナス元金 / m`
5. **月々支払合計**: `通常月返済額 + 管理費 + 修繕積立費 + その他費用`
   - 管理費・修繕積立費・その他費用は月額固定費としてそのまま加算する
   - 構成要素のいずれかが算出不能な場合の扱い（合計を '—' とするか、入力済み分のみ加算するか）は受け入れ基準で明示する（下記 Sprint 1 参照）

## スプリント計画

### Sprint 1: 基盤構築 + 計算ロジック + 単一物件シミュレーション
**ゴール:** 1つの物件について、入力値からローン支払額をリアルタイムに計算・表示できる状態にする。計算ロジックを独立したモジュールとして完成させる。

**機能:**
- [ ] プロジェクトセットアップ: アプリが起動し、空のシミュレーション画面が表示される
- [ ] ローン入力フォーム: 8項目（物件価格・諸経費・金利・ローン期間・ボーナス元金・管理費・修繕積立費・その他費用）を入力できる。各項目は空欄を許容する
- [ ] 計算ロジックモジュール: 上記「計算ロジック仕様」に従い、通常月返済額・ボーナス払い・月々支払合計・資金構成を算出する純粋関数群を実装する
- [ ] 金利0%対応: 金利が0のとき、ゼロ除算を起こさず `元金 / 返済回数` で算出する
- [ ] null安全表示: 未入力・NaN・Infinity の値は '—' で表示する
- [ ] リアルタイム再計算: 入力変更のたびに結果が即時更新される
- [ ] 計算結果表示: 月々支払合計と、その内訳（通常月返済額・ボーナス払い月額換算・管理費・修繕積立費・その他費用）を表示する

**受け入れ基準:**
- ユーザーが物件価格3,000万円・諸経費200万円・金利1.0%・期間35年・ボーナス元金0円を入力すると、通常月返済額が約90,000円台で表示されること（元利均等の理論値と一致）
- 金利を0%にしても結果が '—' やエラーにならず、`元金 ÷ 返済回数` の値が表示されること
- 物件価格を空欄にすると、月々支払合計が '—' と表示されること
- 金利欄に文字や負数など計算不能な値が入っても、アプリがクラッシュせず該当結果が '—' 表示になること
- いずれかの入力値を変更すると、ボタンを押さずに計算結果が更新されること
- 金額が3桁区切りで表示されること
- 月々支払合計が「通常月返済額＋管理費＋修繕積立費＋その他費用」と一致すること（手計算で検証可能）

### Sprint 2: 複数物件管理
**ゴール:** 複数の物件を登録・編集・削除・切り替えでき、それぞれが独立した入力値と計算結果を保持する状態にする。

**機能:**
- [ ] 物件の追加: 「物件を追加」操作で新規物件が作成され、空の入力フォームが用意される
- [ ] 物件の切り替え: 登録済み物件の一覧から編集対象を選択し、その物件の入力値・計算結果に切り替わる
- [ ] 物件名の編集: 各物件に名前を付与・変更でき、一覧や見出しに反映される
- [ ] 物件の削除: 物件を削除でき、一覧から消える。削除後も他の物件の値が保持される
- [ ] データ永続化: ページをリロードしても登録済みの全物件データが保持される

**受け入れ基準:**
- ユーザーが物件を2件追加し、それぞれ異なる物件価格を入力すると、切り替えたときに各物件の値が混ざらず正しく表示されること
- 物件名を「Aマンション」に変更すると、一覧と見出しに「Aマンション」が表示されること
- 物件を削除すると一覧から消え、残った物件の入力値・計算結果が保持されていること
- 物件が1件のときと複数件のときで、いずれもアプリが正常に動作すること
- ページをリロードしても、追加した物件・入力値・物件名が失われないこと
- 全物件を削除しても、アプリがクラッシュせず新規追加が可能な状態に戻ること

### Sprint 3: 物件比較表示 + UI仕上げ
**ゴール:** 登録済み物件を横並びで比較でき、あらゆる画面幅・入力状態で破綻しない完成された UI にする。

**機能:**
- [ ] 物件比較表示: 複数物件の主要指標（物件名・月々支払合計・通常月返済額・ボーナス払い月額換算・固定費内訳・総借入額）を一覧で横並び比較できる
- [ ] 比較時の空欄処理: 比較表示でも未入力・計算不能項目は '—' で表示される
- [ ] レスポンシブ対応: スマホ（縦長）では縦スクロール、PC では横並びなど、各画面幅で読みやすくレイアウトされる
- [ ] 資金構成サマリー: 総借入額（物件価格＋諸経費）と通常返済元金（ボーナス元金控除後）を各物件で確認できる
- [ ] UI仕上げ: 入力フォーム・結果・比較表の視認性、余白、ラベル整合性を整える

**受け入れ基準:**
- ユーザーが3件の物件を登録すると、3件の月々支払合計が横並びで同時に比較できること
- 比較表示中、入力途中で空欄のある物件の該当セルが '—' と表示され、他物件の表示が崩れないこと
- スマートフォン幅（375px 程度）で、横スクロールや要素のはみ出しなく全機能が利用できること
- PC 幅（1280px 程度）でも比較表が読みやすくレイアウトされること
- 金利0%の物件と金利ありの物件を同時に比較しても、両方が正しく表示されること
- 比較表示と個別編集画面を行き来しても、データが一致して保持されていること

### Sprint 4: クラウド同期（Firestore）+ 同期コード共有 + 入力補助
**ゴール:** ログイン不要のまま、同期コード（Sync ID）方式でデータを Firestore に保存し、複数端末で同じ物件データを共有できる状態にする。あわせて諸経費入力を補助するクイック入力を追加する。

**機能:**
- [ ] クラウド同期（Firestore）: 物件データの保存先を localStorage から Firestore に切り替え、どの端末からでも同じデータにアクセスできる（Must）
- [ ] 同期コード（Sync ID）の自動発行: 初回アクセス時に一意な同期コード（UUID）を生成し、端末内に記憶する。この同期コードがクラウド上のデータの保存キーとなる（Must）
- [ ] 同期コードの表示・コピー: 現在の同期コードを画面に表示し、ワンタップ/ワンクリックでコピーできる（Must）
- [ ] 同期コードによるデータ引き継ぎ: 別端末で既存の同期コードを入力すると、その同期コードに紐づく物件データを読み込んで共有できる（Must）
- [ ] 物件名のクラウド保存: 物件名の編集結果が他の入力値と同様に Firestore に保存され、同期される（Must）
- [ ] 諸経費の自動入力ボタン: 諸経費（miscCost）入力欄の横に「物件価格の5%」を自動入力するボタンを設置し、押下すると諸経費欄に物件価格×0.05 がセットされる（Must）
- [ ] 同期状態の表示: 保存中/保存済みなど、クラウド同期の状況がユーザーに分かるよう表示する（Should）
- [ ] オフライン・通信失敗時の堅牢性: 通信に失敗してもアプリがクラッシュせず、入力・計算が継続できる（Should）

**Firestore データ構造の概要:**
- 保存単位は同期コードごとのドキュメント `sessions/{syncId}` とする
- 当該ドキュメントの `state` フィールドに、アプリ全体の状態 `{ properties, activeId }` を保存する
  - `properties`: 全物件の配列（各物件の8項目の入力値・物件名・物件ID を含む）
  - `activeId`: 現在編集対象として選択されている物件のID
- 端末側は同期コード（syncId）のみをローカルに保持し、物件データ本体はクラウド（Firestore）を正とする
- ※ Firestore のセキュリティルール・SDK 設定・データ書き込みの実装方式は Generator が決定する

**受け入れ基準:**
- 初回アクセス時に同期コードが自動生成され、画面上に表示されること
- 物件を追加・編集すると、その内容が Firestore（`sessions/{syncId}/state`）に保存されること（再読み込みしても同じデータが復元される）
- 表示された同期コードをコピーでき、別端末/別ブラウザでその同期コードを入力すると、同じ物件一覧・入力値・物件名・選択中物件が表示されること
- 物件名を変更すると、その物件名がクラウドに保存され、同じ同期コードを使う別端末でも変更後の物件名が表示されること
- 諸経費欄の横の自動入力ボタンを押すと、諸経費欄に「物件価格 × 5%」の値がセットされ、月々支払合計などの計算結果が即座に再計算されること
- 物件価格が未入力の状態で諸経費自動入力ボタンを押しても、アプリがクラッシュしないこと（諸経費は0または未入力相当として扱う）
- 通信が失敗しても入力・計算が継続でき、復帰後に再び同期できること
- 既存の Sprint 1〜3 の全機能（計算・複数物件管理・比較表示・レスポンシブ）が引き続き正常に動作すること（回帰がないこと）

### Sprint 5: 単一共有ドキュメント方式への移行（localStorage・同期コード廃止）
**ゴール:** 識別子・ログイン・同期コードを一切不要とし、すべての利用者が同じ Firestore ドキュメント（`shared/main`）を共有する。ブラウザを開くだけで自動的に最新データを読み込み、編集すると全員にリアルタイムで反映される状態にする。localStorage への依存を完全に排除する。

**背景（Sprint 4 からの変更点）:**
Sprint 4 では同期コード（Sync ID）方式で `sessions/{syncId}` にデータを分離保存し、端末側に syncId を localStorage で保持していた。Sprint 5 ではこの方式を撤廃し、全利用者が単一の共有ドキュメントを参照する「共有ホワイトボード」型に変更する。識別・分離の概念をなくし、誰がアクセスしても同じデータが見える状態にする。

**機能:**
- [ ] localStorage の完全廃止: アプリ内のすべての localStorage への読み書き（旧 `loadState` / `saveState` の呼び出し、syncId の保存・読み出しを含む）を削除する。アプリは localStorage に一切アクセスしない（Must）
- [ ] 同期コード機能の削除: 同期コードの生成・表示・コピー・入力・切り替えを担う UI（SyncBar コンポーネント）を画面から完全に削除する（Must）
- [ ] 同期コード関連ロジックの削除: `getSyncId` / `saveSyncId` 関数、および同期コードに依存する処理経路をコードベースから削除する（Must）
- [ ] 単一共有ドキュメントへの統一: データの保存先を `shared/main` 固定ドキュメントに統一する。識別子・パラメータによるドキュメント分岐を行わない（Must）
- [ ] 起動時の自動読み込み: ブラウザでアプリを開くと、ユーザー操作（コード入力・ログイン等）なしに `shared/main` から最新データを自動で読み込み、画面に反映する（Must）
- [ ] リアルタイム購読の維持: Firestore の `onSnapshot` による購読を維持し、他の利用者の編集が自動的に画面へ反映される（Must）
- [ ] 編集内容の共有ドキュメントへの保存: 物件の追加・編集・削除・物件名変更・選択中物件の切り替えが、すべて `shared/main` に保存され、全利用者間で共有される（Must）
- [ ] オフライン時の読み取り専用モード: 通信が切断（オフライン）された場合、画面上部に「オフライン中・読み取り専用」を示すバナーを表示し、すべての編集操作（物件の追加・編集・削除・物件名変更・選択中物件の切り替え・諸経費5%ボタン）を無効化（ブロック）する。閲覧と計算結果の表示は継続できる（Must）
- [ ] オンライン復帰時の編集再開: 通信が復帰すると読み取り専用バナーが消え、編集操作が再び有効になり、`shared/main` との同期が再開される（Must）
- [ ] オフライン時の堅牢性（localStorage 非依存）: オフライン・通信失敗時もアプリがクラッシュせず、直前に読み込んだデータの閲覧・計算結果表示が継続できる。この堅牢性を localStorage を使わずに実現する（Should）

**Firestore データ構造の概要（Sprint 5）:**
- 保存単位は単一の固定ドキュメント `shared/main` とする（`sessions/{syncId}` 方式は廃止）
- ドキュメントは以下のフィールドを持つ:
  - `properties`: 全物件の配列（各物件の8項目の入力値・物件名・物件ID を含む）`Property[]`
  - `activeId`: 現在選択中の物件のID `string | null`
- 端末側に保持する識別子（syncId 等）は存在しない。データ本体はすべてクラウド（`shared/main`）を正とする
- ※ Firestore のセキュリティルール・初期ドキュメント生成・書き込みの実装方式は Generator が決定する

**受け入れ基準:**
- アプリ内のソースコードに localStorage への読み書き（`localStorage.getItem` / `localStorage.setItem` 等）が一切存在しないこと（grep で検出されないこと）
- 画面上に同期コードの表示・コピー・入力欄（SyncBar）が一切表示されないこと
- `getSyncId` / `saveSyncId` 関数がコードベースに存在しないこと
- 何の操作もせずブラウザでアプリを開くと、識別子入力やログインを求められることなく、`shared/main` の物件データが自動で表示されること
- ある端末（ブラウザA）で物件を追加・編集すると、別端末（ブラウザB）で新規にアプリを開いたときに同じ物件データが表示されること（全員が同じデータを共有していること）
- ブラウザB を開いたまま、ブラウザA で物件名や入力値を変更すると、ブラウザB の表示にもリアルタイムで反映されること（onSnapshot が機能していること）
- 物件の追加・削除・物件名変更・選択中物件の切り替えがいずれも `shared/main` に保存され、再読み込み後も保持されること
- 通信を遮断（オフライン）にすると、画面上部に「オフライン中・読み取り専用」を示すバナーが表示されること
- オフライン中は、物件の追加・編集・削除・物件名変更・選択中物件の切り替え・諸経費5%ボタンといった編集操作がすべて無効化（ブロック）され、データが変更されないこと
- オフライン中でも、直前に読み込んだ物件データの閲覧と計算結果の表示は継続でき、アプリがクラッシュしないこと
- 通信を復帰させると読み取り専用バナーが消え、編集操作が再び可能になり、`shared/main` との同期が再開されること
- ブラウザの localStorage を手動で全消去した直後にアプリを開いても、データが失われず `shared/main` から正しく表示されること（localStorage に依存していないことの確認）
- 既存の Sprint 1〜4 の機能のうち、計算ロジック・複数物件管理・物件比較表示・諸経費5%自動入力ボタンが引き続き正常に動作すること（回帰がないこと。ただし同期コード共有・localStorage 永続化は Sprint 5 で意図的に廃止された機能であり、回帰対象外とする）

### Sprint 6: 物件メタデータ（URL・メモ・物件種別・アクセス）
**ゴール:** ローン計算には影響しない「物件のメタ情報」（物件URL・テキストメモ・新築/中古区分・駅徒歩時間）を各物件に保存・表示できるようにする。これらは比較検討時の判断材料として、入力フォームと比較表の両方で確認できる状態にする。

**背景・位置づけ:**
これまでの Sprint 1〜5 は「月々の支払額」という金額計算に特化していたが、実際の物件比較では金額以外の属性（立地・築年数・物件ページのリンク・個人的なメモ）も重要な判断材料となる。Sprint 6 では計算ロジックには一切手を加えず、物件に付随するメタデータを追加する。新フィールドはすべて任意入力（空欄・未選択を許容）とし、未入力時も計算・表示が破綻しないことを前提とする。

**機能:**
- [ ] 物件URLの保存・表示: 物件情報ページへのリンクをテキスト入力欄で保存できる。保存された URL はリンク（別タブで開ける `<a href target="_blank">`）として表示される（Must）
- [ ] テキストメモの保存・表示: 物件に関する自由記述メモを複数行テキストエリアで入力・保存できる。改行を含む内容が保持される（Must）
- [ ] 新築／中古の選択: 「新築」「中古」の2択をトグルボタンで選択でき、未選択状態も許容する。選択内容が保存・表示される（Must）
- [ ] 駅徒歩時間の入力: 最寄り駅からの徒歩時間を整数（分単位）で入力でき、空欄も許容する。入力値が保存・表示される（Must）
- [ ] 「物件情報」セクションの設置: 上記4項目を、既存の金額入力フィールド（ローン計算用）とは区別できる「物件情報」セクションとして LoanForm 内に配置する。ローン計算用フィールドとメタ情報フィールドの区別が視覚的にわかること（Must）
- [ ] 比較表への反映: 比較表（CompareTable）に「新築/中古」「駅徒歩時間」「物件URL（短縮表示＋リンク）」の行を追加し、各物件のメタ情報を横並びで比較できる（Must）
- [ ] 共有ドキュメントへの保存: 新フィールドは各物件オブジェクトの一部として `shared/main` の `properties` 配列に保存され、全利用者間で共有・リアルタイム同期される（Must）
- [ ] 既存データとの後方互換: これらのフィールドを持たない既存物件データを読み込んだ場合も、デフォルト値で正しく扱われ、アプリがクラッシュしない（Must）

**実装上の要件（Generator 向け）:**
- `Property` 型（`src/lib/storage.ts`）に以下のフィールドを追加する:
  - `url?: string` — 物件情報ページの URL
  - `memo?: string` — 自由記述メモ（複数行）
  - `buildingAge: 'new' | 'used' | null` — 新築/中古/未選択
  - `walkMinutes?: number` — 駅徒歩時間（分・整数、空欄許容）
- `createProperty()` の初期値に追加する: `url: ''`, `memo: ''`, `buildingAge: null`, `walkMinutes: undefined`
- `normalizeState()` で、既存物件にこれらのフィールドが存在しない場合のデフォルト値（`url: ''`, `memo: ''`, `buildingAge: null`, `walkMinutes: undefined` 相当）を補完する
- `LoanForm.tsx` に4項目の入力欄を追加する。入力ハンドリングは LoanForm の `FormValues` 型に組み込む方式でも、`Property` を直接更新する方式でも、App.tsx で別ハンドラを設ける方式でもよい（Generator が判断）
  - 物件URL: テキスト入力（プレフィックス `¥` なし）
  - テキストメモ: 複数行テキストエリア
  - 新築/中古: 2択トグルボタン（未選択を許容）
  - 駅徒歩時間: 数値入力（単位「分」、整数）
- `CompareTable.tsx` に「新築/中古」「駅徒歩時間」「物件URL（短縮表示＋リンク）」の行を追加する
- 物件URL は比較表・フォームいずれの表示箇所でもクリックで別タブで開けるリンクとして描画する
- これらのメタフィールドはローン計算（計算ロジック仕様）に一切影響を与えないこと
- Firestore のスキーマ変更は不要。既存の `properties` 配列内の各物件オブジェクトにフィールドが増えるだけで、追記保存される（`shared/main` 構造は維持）

**受け入れ基準:**
- 物件URL欄に URL を入力すると保存・表示され、表示されたリンクをクリックすると別タブで該当 URL が開くこと
- テキストメモ欄に複数行の文章を入力すると保存・表示され、改行を含む内容が保持されること
- 「新築」「中古」のトグルを切り替えられ、選択した内容が保存・表示されること。未選択の状態も保持できること
- 駅徒歩時間を整数（分）で入力すると保存・表示されること。空欄のままでもアプリが破綻しないこと
- 比較表に「新築/中古」「駅徒歩時間」「物件URL（短縮表示＋リンク）」の行が追加され、複数物件のメタ情報を横並びで比較できること。比較表の物件URLもクリックで別タブで開けること
- 上記の値が `shared/main` に保存され、ブラウザをリロードしても保持されること。別端末/別ブラウザで開いても同じメタ情報が表示されること（共有・リアルタイム同期されること）
- これらのメタフィールドの入力・変更によって、月々支払合計や各種計算結果が変化しないこと（計算への非干渉）
- これらのフィールドを持たない既存物件データ（Sprint 5 までに作成されたデータ）を読み込んでも、デフォルト値で正しく表示され、アプリがクラッシュしないこと
- 既存の Sprint 1〜5 の機能（計算ロジック・複数物件管理・物件比較表示・諸経費5%自動入力・`shared/main` 単一共有同期・オフライン読み取り専用モード）に回帰がないこと

### Sprint 7: 夫婦優先度比較
**ゴール:** 夫と妻がそれぞれ「住宅購入における優先条件」を 5 段階で入力し、2人の優先度を並べて視覚的に比較できる独立した機能を追加する。ローン計算・物件管理とは別タブとして実装し、既存機能に一切影響を与えない。

**背景・位置づけ:**
これまでの Sprint 1〜6 は「物件そのものの条件（金額・メタ情報）」を扱ってきたが、住宅購入は夫婦の合意形成プロセスでもある。何を重視するか（駅近・広さ・価格・静かさ等）は夫婦間で食い違いやすく、その差を可視化することは購入判断の重要な助けとなる。Sprint 7 では既存のローン計算機能とは独立した「夫婦優先度」機能を追加し、夫・妻それぞれの優先度入力と、2人の優先度を並べた比較ビューを提供する。この機能はローン計算ロジック・物件データ（`properties` / `activeId`）に一切干渉しない別系統のデータとして扱う。

**優先条件の項目（固定候補 + カスタム追加）:**
以下を固定の初期候補とする（Generator は表示順・キー設計を実装時に決定してよいが、この11項目を初期表示すること）:
1. 駅からの距離
2. 物件価格
3. 広さ・間取り
4. 新築かどうか
5. 管理費・修繕積立費
6. 日当たり・採光
7. 収納の多さ
8. 駐車場の有無
9. 学校・保育園の近さ
10. 買い物の便利さ
11. 静かさ・治安

- 各条件には 5 段階の優先度を設定する（1=気にしない 〜 5=絶対必要。★1〜5 と対応）。
- 上記固定11項目に加え、ユーザーが任意のラベルでカスタム項目を追加できる（追加項目も夫・妻それぞれ 5 段階で入力可能）。

**機能:**
- [ ] 「夫婦優先度」タブの追加: 既存のローン計算系ナビ（入力 / 結果 / 比較）とは別に、トップレベルのナビ/タブとして「夫婦優先度」を追加する。既存タブとの行き来でローン計算側の状態が失われないこと（Must）
- [ ] 夫・妻・比較の3サブタブ: 「夫婦優先度」ページ内に「夫」「妻」「比較」の3サブタブを設け、切り替えられる（Must）
- [ ] 夫タブの優先度入力: 固定11項目（＋カスタム項目）を縦並びで表示し、各条件の優先度を 1〜5 で入力・変更できる（★スライダーまたは★ボタン等、デザイン仕様に沿った UI）（Must）
- [ ] 妻タブの優先度入力: 夫タブと同一の項目群に対し、妻の優先度を 1〜5 で独立に入力・変更できる（夫の値と混ざらないこと）（Must）
- [ ] カスタム項目の追加: 任意のラベルで条件項目を追加でき、夫・妻の両タブと比較タブに反映される（Must）
- [ ] カスタム項目の削除: 追加したカスタム項目を削除でき、削除後も固定11項目および他のカスタム項目の値が保持される（Should）
- [ ] 比較タブの横並び表示: 比較タブで、各条件について夫と妻の優先度を横並び（バー or テーブル形式）で視覚的に比較できる（Must）
- [ ] 差分の強調表示: 夫と妻の優先度の差が大きい項目（`|husband - wife| >= 2`）を視覚的に強調表示する（色・アイコン・ラベル等で「意見が分かれている」ことが一目でわかる）（Must）
- [ ] 共有ドキュメントへの保存: 夫・妻の優先度およびカスタム項目が `shared/main` の `coupleConditions` フィールドに保存され、リロード後も保持される。全利用者間で共有・リアルタイム同期される（Must）
- [ ] 既存データとの後方互換: `coupleConditions` フィールドを持たない既存ドキュメントを読み込んだ場合も、デフォルト値（全項目未入力相当）で正しく初期化され、アプリがクラッシュしない（Must）
- [ ] オフライン時の挙動整合: Sprint 5 のオフライン読み取り専用モードと整合させ、オフライン中は優先度の入力・変更・カスタム項目の追加削除を無効化（ブロック）し、閲覧と比較表示は継続できる（Should）

**データ構造の概要（Sprint 7）:**
- `shared/main` ドキュメントに新たに `coupleConditions` フィールドを追加する。既存の `properties` / `activeId` フィールドには影響しない:
  ```
  coupleConditions: {
    husband: { [conditionKey]: number }   // 各条件キーに対する優先度 1〜5
    wife:    { [conditionKey]: number }    // 各条件キーに対する優先度 1〜5
    customItems: { key: string; label: string }[]  // カスタム追加項目の定義
  }
  ```
- 固定11項目は安定したキー（例: `station`, `price`, `size` 等）で `husband` / `wife` に保存する。キーの命名は Generator が決定してよい。
- カスタム項目は `customItems` にキーとラベルを登録し、その優先度値は固定項目と同じく `husband` / `wife` に同じキーで保存する。
- 未入力の条件は値なし（または既定値）として扱い、比較時は「未入力」相当の表現にする（クラッシュしないこと）。
- ※ Firestore のセキュリティルール・既定ドキュメント生成・書き込み方式は Generator が決定する。`shared/main` 単一共有ドキュメント方式（Sprint 5）は維持する。

**受け入れ基準:**
- トップレベルナビ/タブに「夫婦優先度」が表示され、クリックすると夫婦優先度ページに遷移できること。既存のローン計算側のタブ（入力 / 結果 / 比較）と相互に行き来できること
- 「夫婦優先度」ページ内に「夫」「妻」「比較」の3サブタブが表示され、切り替えられること
- 夫タブで各条件（固定11項目）の優先度を 1〜5 で入力・変更でき、変更が即座に反映されること
- 妻タブで各条件の優先度を 1〜5 で独立に入力・変更でき、夫の入力値と混ざらないこと
- 任意のラベルでカスタム項目を追加すると、夫タブ・妻タブ・比較タブのすべてにその項目が表示され、夫・妻それぞれ優先度を入力できること
- 比較タブで、各条件について夫と妻の優先度が横並び（バー or テーブル）で確認できること
- 夫と妻の優先度の差が 2 以上（`|husband - wife| >= 2`）の項目が、差の小さい項目と区別して視覚的に強調表示されること
- 入力した夫・妻の優先度およびカスタム項目が `shared/main` の `coupleConditions` に保存され、ページをリロードしても保持されること。別端末/別ブラウザで開いても同じ優先度が表示され、一方の編集が他方にリアルタイム反映されること
- `coupleConditions` フィールドを持たない既存データ（Sprint 6 までに作成されたデータ）を読み込んでも、デフォルト値で正しく初期化され、アプリがクラッシュしないこと
- 夫婦優先度機能の操作によって、ローン計算・物件管理・物件比較・メタデータ等の既存データや表示が変化しないこと（別系統データであることの確認）
- 既存の Sprint 1〜6 の機能（計算ロジック・複数物件管理・物件比較表示・諸経費5%自動入力・`shared/main` 単一共有同期・オフライン読み取り専用モード・物件メタデータ）に回帰がないこと

## UI/UX 要件

**画面構成（単一ページアプリ想定）:**
- 物件リスト（サイドバーまたはタブ）: 登録済み物件の一覧。選択・追加・削除・名称変更の起点
- 入力フォーム領域: 選択中物件の8項目を入力。各項目はラベル付き数値入力で、空欄可
- 結果サマリー領域: 月々支払合計を大きく強調表示し、その下に内訳（通常月返済額・ボーナス払い・固定費）と資金構成を表示
- 比較ビュー: 全物件の主要指標を横並び（テーブルまたはカード）で表示

**主要ユーザーフロー:**
1. 初回訪問 → 物件1件目が用意済み、または「物件を追加」で開始
2. 入力 → 入力と同時に結果がリアルタイム更新
3. 物件追加 → 別条件を入力 → 比較ビューで横並び確認
4. 不要物件の削除・物件名の整理

**UI コンポーネント:**
- 数値入力フィールド（プレースホルダー・単位ラベル付き、3桁区切り表示）
- 物件切り替え用のリスト/タブ
- 結果カード（合計強調 + 内訳明細）
- 比較テーブル/比較カード群

## 非機能要件
- **パフォーマンス:** 入力変更から結果更新までが体感即時（再計算で UI がもたつかないこと）
- **堅牢性:** どの入力欄に不正値・空欄が入ってもアプリがクラッシュしないこと
- **レスポンシブ:** スマホ・タブレット・PC の主要画面幅で利用可能
- **アクセシビリティ:** 各入力欄にラベルが関連付けられ、キーボード操作で入力・物件切り替えが可能
- **データ保持:** ブラウザ上でデータが永続化され、サーバ送信を前提としない（ローカル完結）

## 制約事項（スコープ外）
- **本仕様の対象は元利均等返済のみ。** 元金均等返済方式は対象外とする
- **変動金利・金利変動シミュレーション・繰上返済は対象外。** 金利は全期間固定の前提で計算する
- **返済予定表（毎月の元金・利息内訳の年次推移）の出力は対象外。** 本アプリは月々負担の比較に特化する
- **税制優遇（住宅ローン控除）・諸費用の自動見積りは対象外。** 諸経費・各種固定費はユーザーが手入力する
- **ユーザーアカウント・ログイン認証は対象外。** クラウド同期・複数端末共有は同期コード（Sync ID）方式で実現し、ユーザー登録・認証は行わない（Sprint 4 参照）
- **同期コードのアクセス制御・暗号化は対象外。** 同期コードを知っていれば誰でも該当データにアクセスできる前提とし、機密情報の保存は想定しない
- **金融機関データ連携・実際の借入手続きは対象外。** 本アプリは試算ツールであり、表示値は概算である
